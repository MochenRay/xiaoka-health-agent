## 一、背景与动机

### 问题

当前个人健康管理存在严重的**数据割裂**：

- **饮食数据**在薄荷健康
- **运动/心率/睡眠**在 Apple Health
- **体检报告**在医院 App 或纸质文件
- **补剂方案**在脑子里
- **药物知识**散落在各种科普文章

没有一个地方把所有健康数据串起来，做跨维度的分析。

### 市场空白

| 功能 | 薄荷健康 | MyFitnessPal | Apple Health | **市场空白** |
|------|---------|-------------|-------------|------------|
| 饮食记录 | ✅ | ✅ | ❌ | — |
| 运动追踪 | ✅ | ✅ | ✅ | — |
| 血检/体检解读 | ❌ | ❌ | ❌ | **无人覆盖** |
| 补剂循证管理 | ❌ | ❌ | ❌ | **无人覆盖** |
| GLP-1 药物决策 | ❌ | ❌ | ❌ | **无人覆盖** |
| 伤病感知运动编程 | ❌ | ❌ | ❌ | **无人覆盖** |
| 跨维度关联分析 | ❌ | ❌ | 部分 | **基本无人覆盖** |
| 数据本地存储 | ❌ | ❌ | ❌ | **无人覆盖** |

### 竞品分析结论

> 不跟薄荷/MyFitnessPal 拼食品数据库（611 条 vs 5000 万条），而是成为**医学驱动的健康分析层**——与现有工具互补，做它们做不了的事。

## 二、产品定位

### 一句话

**小卡是一个开源的 AI 健康教练 Agent，专注医学级分析——血检解读、补剂管理、GLP-1 决策、跨维度健康趋势追踪，数据本地存储。**

### 核心理念

1. **医学分析层，不是热量计数器**——站在薄荷/Apple Health 的肩膀上做深度分析
2. **数字先行，说教在后**——先给数据表格，再给建议
3. **隐私优先**——所有健康数据本地存储。注：使用云端模型（如 qwen3.5-plus）时，对话内容会发送给模型提供商
4. **临床严谨**——引用来源可追溯，不编造数据
5. **开源可定制**——Agent 包开源，任何人可 clone 部署

### 不做的事

- ❌ 不做食品条码扫描（薄荷做得更好）
- ❌ 不做实时运动追踪（Apple Watch 做得更好）
- ❌ 不做社交/排行榜功能
- ❌ 不提供医疗诊断或处方（始终声明"请咨询医生"）

## 三、目标用户

### 主要用户画像

| 画像 | 特征 | 核心需求 |
|------|------|---------|
| **有技术背景的减重者** | 会用 CLI / Telegram bot，注重数据精度 | 打通饮食+运动+体检的割裂，精确追踪 |
| **GLP-1 药物使用者** | 正在或考虑使用司美格鲁肽/替尔泊肽 | 决策支持、副作用监测、停药管理 |
| **健身教练** | 需要为客户解读体检报告、规划补剂 | 专业知识库 + 结构化输出 |
| **隐私敏感者** | 不信任云存储健康数据 | 100% 本地存储 |

### 使用场景举例

| 场景 | 交互方式 |
|------|---------|
| "午饭吃了板烧鸡腿堡+零度可乐" | 文字 → 热量核算 → 记入日志 |
| 发一张薄荷健康今日摘要截图 | 图片 → Vision 识别 → 结构化存储 |
| 发一张体检报告照片 | 图片 → 逐项解读 → 标注异常 → 存入体检档案 |
| "肌酸和鱼油能一起吃吗？" | 问答 → 查补剂知识库 → 回答+来源 |
| "这周情况怎么样？" | 触发周报 → 汇总趋势+建议 |
| 体重停滞两周不动 | Agent 展示其他维度变化 → 策略调整建议 |

## 四、Agent 身份：小卡

### 基本信息

- **名字**：小卡（"卡路里"的卡）
- **定位**：你身边一个靠谱的、有专业知识的朋友
- **与其他 Agent 的区分**：不是你带的小孩（小雷），不是信息官（Herald），不是收藏助手（Pocket）——是一个懂健康的温和教练

### 性格设定

- **温和顾问型**：专业但说人话，不端架子
- **偶尔幽默**：不是段子手，但会在合适的时候轻松一下
- **有实质的回应**：每次交互都给出有信息量的反馈，不说"已记录"这种废话
- **不催不逼**：用户沉默时不发催促消息，下次来时正常回应

### 三条核心行为规则

这三条规则基于用户心理特征提炼，写入 Agent 人格设定：

1. **回应要有实质内容**
   - ❌ "已记录你的午餐"
   - ✅ "午餐 520kcal，蛋白质 25g 不错。今天还剩 1230kcal 预算，晚上碳水可以少一点"

2. **平台期不盖盖子**
   - 体重停滞时不说"继续加油"
   - 展示其他维度的变化（体脂、腰围、运动表现）
   - 提供策略调整选项（diet break、调整训练、重新计算 TDEE）
   - 核心：**不让视野被封闭**

3. **沉默时不催**
   - 用户几天不发消息 → 不发催促/提醒
   - 用户主动回来时 → 正常回应，不提"你好久没来了"
   - 唯一例外：Cron 任务按计划执行（周报/月报/前日汇总），但不额外催

## 五、系统架构

### 整体分层

```
┌─────────────────────────────────────────────┐
│                  用户交互层                    │
│         文字 / 图片 / 对话 (Telegram等)        │
└──────────────────┬──────────────────────────┘
                   ▼
┌─────────────────────────────────────────────┐
│                  Agent 层                     │
│    人格设定 (agent.md) + 技能路由 (SKILL.md)   │
└──────────────────┬──────────────────────────┘
                   ▼
┌─────────────────────────────────────────────┐
│                知识引擎层                      │
│   references/ (知识库) + cn-food-db.json      │
└──────────────────┬──────────────────────────┘
                   ▼
┌─────────────────────────────────────────────┐
│                数据持久层                      │
│  workspace/ (Markdown日志 + JSON结算 + 体检)   │
└──────────────────┬──────────────────────────┘
                   ▼
┌─────────────────────────────────────────────┐
│                自动化层                        │
│    Cron: 零点结算 / 周报 / 月报 / 前日汇总      │
└─────────────────────────────────────────────┘
```

### 模型依赖

- **主要依赖**：大语言模型的基础能力（文本理解、图片识别、营养知识、数学计算）
- **推荐模型**：qwen3.5-plus（原生多模态，中文优秀）、kimi-k2.5（原生多模态）、Claude（推理质量最高）
- **最低要求**：支持多模态（图片输入）的模型。纯文本模型可用但无法处理截图

## 六、功能模块设计

### 6.1 数据输入模块

#### A1. 饮食记录

**触发**：用户发送食物相关的文字描述或图片

**处理流程**：
```
用户输入 → 读取当日已有日志（计算已有累计）
         → 识别食物种类和份量
         → 查询食物数据（三层架构）
         → 计算热量和宏量营养素
         → 对比每日目标，计算剩余预算
         → 追加到当日 Markdown 日志
         → 返回结构化反馈（含今日累计）
```

**食物数据三层架构**：
**查询优先级：第一层 > 第二层 > 第三层**（用户自建的精确数据优先于通用数据）

1. **第一层：用户自建常吃食物库**（品牌产品、外卖、奶茶，JSON）——通过拍摄食物包装上的营养成分表照片，Agent 自动识别并录入。精度最高，因为是实际产品的包装标注
2. **第二层：中国食物成分表**（1,677 条基础食材，权威 JSON）——通用食材的平均值，适合无品牌的基础食材（如"鸡胸肉""白米饭"）
3. **第三层：AI 模型估算**（模型内置知识，标注"AI 估算"）

**输出格式**（强制）：
```
🍽️ 午餐记录

| 食物 | 份量 | 热量 | 蛋白质 | 碳水 | 脂肪 |
|------|------|------|--------|------|------|
| 板烧鸡腿堡 | 1个 | 390 | 25g | 35g | 16g |
| 零度可乐 | 1杯 | 0 | 0g | 0g | 0g |
| **小计** | | **390** | **25g** | **35g** | **16g** |

📊 今日累计：850 / 1750 kcal（剩余 900）
🥩 蛋白质：62 / 150g（还差 88g）
```

#### A2. 体重打卡

**触发**：用户发送体重数字

**处理流程**：
```
用户输入 "104.5" → 记录体重+日期
                  → 计算 7 天均值
                  → 计算 30 天趋势
                  → 对比目标轨迹
                  → 异常检测（单日变化 >2kg 则询问确认）
                  → 追加到 Markdown 日志
                  → 返回趋势反馈
```

#### A3. 体检录入

**触发**：用户发送体检报告照片或文字数据

**处理流程**：
```
用户输入 → 识别体检指标（Vision 或文字解析）
         → 查 medical-markers 知识库
         → 逐项标注：正常 / 偏高 / 偏低 / 需关注
         → 标注临床意义（通俗解释）
         → 与历史数据对比（如有）
         → 存入 medical/ 目录（单独归档）
         → 附免责声明
```

#### A4. 运动数据

**触发**：用户发送运动描述、Apple Health 截图

**处理流程**：
```
用户输入 → 识别运动类型+时长+消耗
         → 如有 Apple Watch 数据优先使用实际消耗
         → 如无设备数据则用 MET 估算
         → 追加到 Markdown 日志
         → 返回反馈
```

#### A5. 补剂记录

**触发**：用户提及补剂相关内容

**处理流程**：
```
用户输入 → 记录补剂名称+剂量
         → 查 supplements 知识库检查交互作用
         → 如与用户当前药物有交互 → 警告
         → 追加到 Markdown 日志
```

#### A6. 睡眠数据

**触发**：用户发送睡眠信息或 Apple Health 睡眠截图

**处理流程**：
```
用户输入 → 记录时长+质量
         → 追加到 Markdown 日志
```

#### A7. 食物库维护

**触发**：用户拍摄食物包装上的营养成分表照片，或说"记住这个食物"

**处理流程**：
```
用户拍营养成分表照片 → Vision 识别表格内容
  → 提取：品名 / 每百克热量 / 蛋白质 / 碳水 / 脂肪
  → 询问确认："XX 品牌 XX 产品，每 100g 含 XXkcal，对吗？"
  → 用户确认后写入 workspace/food-library/my-foods.json
  → 下次提到该食物时直接从库中查询
```

#### A0. Profile 初始化（首次使用）

**触发**：Agent 检测到 `config/profile.md` 不存在

**处理流程**：
```
首次对话 → 检测 profile.md 是否存在
  → 不存在 → 进入引导式问答：
    1. 身高（cm）
    2. 体重（kg）
    3. 年龄
    4. 性别
    5. 目标（减脂/增肌/维持）
    6. 目标体重
    7. 活动水平（久坐/轻度活动/中度活动/高度活动）
    8. 伤病/过敏（可选）
    9. 当前用药（可选）
  → 自动计算 BMR/TDEE/热量目标/蛋白质目标
  → 生成 config/profile.md + config/goals.md
  → 创建 workspace/ 目录结构
  → "准备好了！你可以开始记录了"
```

**中断恢复**：用户引导中途退出 → 半成品 profile.md 保留 → 下次回来检查字段完整性 → 询问"继续完成还是先跳过？"

**Profile 更新**（用户主动发起）：
```
用户："我想把目标体重从 85 改成 87.5"
  → Agent 更新 profile.md / goals.md 中的对应字段
  → 自动重算受影响的计算值（热量目标、宏量目标、预计达标时间）
  → 展示变化影响："目标调整为 87.5kg，
     每日热量目标 1750→1820kcal，预计达标提前至 7 月中"
  → 确认后写入
```

**goals.md 中的可选开关**：
- `exercise_counts_as_budget`: 运动消耗是否计入剩余热量预算（默认：否）

| 规则 | 阈值 | 动作 |
|------|------|------|

## 七、数据架构

### 7.1 双轨存储：Markdown + JSON

- **白天**：每次交互追加到当日 Markdown 日志（`logs/YYYY-MM/DD.md`）
- **零点**：Cron 任务将 Markdown 结算为结构化 JSON（`data/YYYY-MM/DD.json`）
- **Markdown 是工作区**（人类可读，实时更新）
- **JSON 是数据库**（机器可读，报告/趋势分析的数据源）

## 八、自动化任务（Cron）

| 任务 | 时间 | 功能 | 备注 |
|------|------|------|------|
| **零点结算** | 每天 00:00 | Cron 触发 Agent → 读当天 Markdown 日志 → 理解内容 → 写入 JSON | 由 AI 模型执行。覆盖式（重复触发幂等）。仅当日有日志时执行 |
| **结算校验** | 每天 00:30 | 读取当日 JSON → 对比 Markdown 中各餐次小计加总 → 偏差超阈值则标记异常 | 防止模型漏餐或算错 |
| **前日汇总** | 每天 08:00 | 读取昨日 JSON → 生成简短汇总推送给用户 | **仅当昨日有日志文件时触发**，无记录则静默。哪怕只有体重打卡也发 |
| **周报** | 周六 20:00 | ①检查本周 JSON 齐全性，缺哪天先补结算 ②汇总趋势 → 生成周报 | 始终生成，标注覆盖率（如"本周记录 5/7 天"）。≥3 天给趋势分析，<3 天仅简要回顾 |
| **月报** | 月末最后一天 20:00 | 同周报逻辑：先补缺 → 汇总 → 生成月报 | 同上 |

## 十一、已知约束与风险

| 约束 | 影响 | 应对 |
|------|------|------|
| **食品数据库规模有限** | 基础食材 1,677 条，品牌食品靠 AI 估算 | 三层架构兜底（自建库 > 成分表 > AI）+ 拍营养成分表录入 + 社区贡献 |
| **Apple Health 数据获取受限** | 无法自动拉取，依赖截图或手动导出 | 截图 Vision 兜底，Health Auto Export 作为进阶选项 |
| **图片识别精度** | 食物照片份量估算误差大 | 标注"AI 估算"，鼓励用户手动修正 |
| **品牌数据会过时** | 奶茶品牌每季度换新品 | 标注更新日期，鼓励用户反馈/PR |
| **非医疗产品** | 体检解读可能被误解为诊断 | 每次附免责声明，明确"请咨询医生" |
| **模型依赖** | 不同模型能力有差异 | 推荐 qwen3.5-plus/kimi-k2.5/Claude，标注最低要求 |
| **云端模型隐私** | 使用云端 API 时对话内容会发送给模型提供商 | 在隐私声明和 README 中说明；本地模型可完全离线 |
| **零点结算依赖模型可用性** | 模型超时或服务不可用时结算失败 | 周报/月报 Cron 自动补结算缺失天数；00:30 校验 Cron 检测异常 |
| **单用户架构** | 当前设计不支持多用户 | 暂不支持，未来如需扩展可按 workspace 隔离 |

## 十二、与 health-coach 的关系

本项目脱胎于 [health-coach](https://github.com/MochenRay/health-coach) skill，但做了以下根本性变更：

| 维度 | health-coach | 小卡 |
|------|-------------|------|
| 形态 | Skill（能力模块） | Agent 包（完整的人） |
| SKILL.md | 叙述体文档（298 行） | 指令式 Workflow（≤200 行上限） |
| 知识库 | 未标注来源，部分数据有误 | 标注来源+验证日期+误差说明 |
| 数据存储 | 自由 Markdown | 双轨：Markdown + JSON 结算 |
| 自动化 | report.sh（macOS 不可用） | AI Cron 任务（结算+校验+报告） |
| 人格 | 无 | 温和教练型，有明确行为规则 |
| 数据验证 | 无 | 异常检测（体重/热量/目标） |
| 开源完整度 | 缺部署指南 | Agent 包 + 部署指南 + .env.example |
